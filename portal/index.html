Pipeline in Practice

Pitgun turns raw data frames into engineered signals through a deterministic Rust pipeline. Sources, processors and sinks are fully declarative: you describe the flow in YAML, Pitgun handles decoding, filtering, scaling, formulas and statistics at F1-style data rates.

manifests/dummy-pitgun.yaml 
# Declarative pipeline configuration
source:
  type: udp
  bind_addr: "0.0.0.0"
  port: 5000

processors:
  - type: channel_filter
    channels: ["nEngine", "rThrottle"]
  - type: scale
    channel: "nEngine"
    factor: 2.0
  - type: formula
    output: "omega_engine"
    ast: "manifests/omega_engine.json" # rpm * 2 * PI / 60
  - type: stats

sink:
  type: console
Run Emulator
$ cargo run -p pitgun-emulator -- --target 127.0.0.1:5000 \
    --input nEngine=datasets/telemetry/nEngine.csv \
    --input rThrottle=datasets/telemetry/rThrottle.csv --pace
Start Engine
$ cargo run -p pitgun-cli -- subscribe --config manifests/dummy-pitgun.yaml_

Why Pitgun’s pipeline is different

Pitgun is not a generic data toy. It is designed from real telemetry workloads: high-frequency signals, tight latency budgets, deterministic behaviour — the kind of constraints you normally see around race cars and test benches.

  • Rust-powered, low-latency, memory-safe processing.
  • Manifest-driven pipelines: declarative, shareable, versioned.
  • Formula engine backed by JSON AST bundles, LLM-ready.
  • Telemetry-native: channels, sampling rates, plausibility checks.
  • Extensible traits for custom processors and sinks.
  • Built and battle-tested with F1-style data constraints in mind.
