Introduction

From Raw Telemetry to Distributed Decision Making

Pitgun began as an F1 telemetry framework and evolved into a Distributed Intelligence Mesh. The architectural pivot retained the high-frequency data handling but added distributed compute and governance layers. The result: a modular Rust framework where each crate solves a specific problem, and the full mesh emerges from their composition.

The Modular Philosophy
  • Gateway as Bridge: Use pitgun-gateway solely to normalize UDP→WebSocket streams.
  • Core as Library: Embed pitgun-core in your Rust app for formula evaluation—no network required.
  • Solver as Grid: Compile pitgun-solver to WASM for browser-based volunteer computing.
  • Authority as Policy: Sign and validate configuration manifests cryptographically.
Why This Exists
  • Legacy telemetry systems are passive—Pitgun makes them active decision engines.
  • Enable Policy as Code: engineers define strategy constraints in signed YAML manifests.
  • Provide a Reference Architecture for edge-native High Performance Computing (HPC).
  • Domain-agnostic: Motorsport reference impl, but applicable to Finance, Energy, IoT.
Transport

HTTP/WebSocket Gateway (Axum-based). UDP multicast & MQTT on the roadmap.

Compute

WASM Solver nodes via wasm-bindgen. Monte Carlo simulations distributed to browser workers.

Security

Authority service (WIP) for manifest signing and Tuning Policy enforcement.

Quick Start
Boot the Mesh

Start the Gateway service, connect Solver nodes, and inject historical data via Replay for immediate testing.

$ cargo run -p pitgun-gateway --release
$ cargo run -p pitgun-replay -- \
    --target 127.0.0.1:8080 \
    --input datasets/telemetry/session.csv

Gateway

Stable

The pitgun-gateway service is the mesh's ingress layer—an Axum-based HTTP server that normalizes incoming telemetry into TelemetryFrame structs. It exposes /beacon (POST) for batch ingestion and /ws for real-time WebSocket streams.

Standalone Usage

Deploy the Gateway without other mesh components as a simple protocol bridge. Connect any WebSocket client to stream normalized frames.

$ PITGUN_GATEWAY_BIND=0.0.0.0:8080 \
  cargo run -p pitgun-gateway --release
Configuration
  • PITGUN_GATEWAY_BIND
    Address to bind (default: 127.0.0.1:8080)
  • PITGUN_GATEWAY_DATA_DIR
    Storage path for telemetry
  • PITGUN_GATEWAY_ALLOW_NON_LOOPBACK
    Enable external binds (set to 1)
Endpoints
  • GET /health ↳ Liveness probe (returns 200 OK)
  • POST /beacon ↳ Batch ingest JSON telemetry payloads
  • GET /ws ↳ Upgrade to WebSocket for real-time streaming
Beacon Payload (JSON)
// POST /beacon
{
  "schema_version": "session-envelope-v1",
  "session_id": 1234567890,
  "sent_at_ms": 1707235200000,
  "batch": {
    "events": [...]
  }
}
WebSocket Messages
  • Text Mode
    JSON SessionEnvelope (same schema as beacon)
  • Binary Mode
    Protobuf EventBatch for low-latency streams
  • Ping / Pong
    Keep-alive heartbeat (auto-handled)
Security

By default, the Gateway binds to 127.0.0.1 only. To expose externally, set PITGUN_GATEWAY_ALLOW_NON_LOOPBACK=1. The Gateway extracts X-Forwarded-For and X-Real-IP headers for audit logging. Backpressure is managed via a bounded ingestion queue (1024 slots)—overflow drops batches with a warning.

Core Engine

Stable

pitgun-core is the brain—a Rust library containing the AST-based Formula Engine, Pipeline orchestration, type converters, and segment aggregation. Use it as a dependency in your own Rust applications.

Library Modules
  • pitgun_core::formula AST parser for expressions (e.g., Power = Torque * RPM)
  • pitgun_core::pipeline Multi-source frame merger with configurable channels
  • pitgun_core::converter Type-safe parameter transformations
  • pitgun_core::segment Lap/session aggregation logic
Embed in Your App
# Cargo.toml
[dependencies]
pitgun-core = "0.22"

Then import the pipeline or formula modules directly—no Gateway or network overhead required.

Manifest-Driven Pipelines

Define derived channels and filters in YAML without recompiling:

# pipeline.yaml
version: v1
pipeline:
  - type: formula
    derived_channels:
      - name: "Power_kW"
        expr: "Torque_Nm * Engine_RPM / 9549.0"
  - type: filter
    whitelist: ["Speed", "Power_kW", "LapTime"]
PipelineConfig Options
  • channel_capacity — Buffer size (default: 4096)
  • max_sources — Max concurrent sources (default: 16)
  • enable_merging — Combine frames from same timestamp
  • merge_window — Time window for merging (default: 1ms)
  • validate_parameters — Check against Registry
Rust API Example
// Multi-source pipeline
let config = PipelineConfig::new()
  .with_channel_capacity(8192)
  .with_merging(Duration::from_millis(2))
  .with_validation();

let mut pipeline = TelemetryPipeline::new(config);
pipeline.add_source(udp_source);
pipeline.add_source(ws_source);

Contract Types

Stable

pitgun-contract defines the shared types and protocols used across all Pitgun components. It acts as the IDL (Interface Definition Language) ensuring type safety between Gateway, Core, Solver, and Authority.

TelemetryFrame

The canonical data structure produced by all sources:

  • session_id: u64 — Unique session identifier
  • sequence: u64 — Monotonic frame counter
  • timestamp_us: i64 — Capture time (μs since epoch)
  • samples: Vec<Sample> — Parameter values
  • events: Vec<Event> — Discrete events
  • lap_number, sector... — Motorsport context
Sample & SampleValue

A parameter reading with type-safe values:

let sample = Sample::new(
  1, // parameter_id
  SampleValue::U16(8500), // RPM
  SignalQuality::Good
);

// SampleValue variants:
U8, U16, U32, I8... F32, F64, Bool
ParameterRegistry

Canonical dictionary of parameter definitions (inspired by ECUBridge PGV):

  • ID → Name/Unit/Type: Lookup by parameter ID
  • Range Validation: Min/max bounds checking
  • Conversions: Raw → engineering units
  • Access Levels: Public, Internal, Restricted
Config Contracts

Signed configuration payloads for Authority:

  • ConfigContractV1 — Signed config with limits
  • SimulationContractV1 — Era, upgrades, policy hash
  • TuningParam — Key-value tuning pairs
  • ContractLimitsV1 — max_hz, max_duration_s
Registry YAML Format
# registries/f1_generic.yaml
version: "1.0"
name: "F1 Generic Registry"
parameters:
  - id: 1
    name: "Engine_RPM"
    unit: "rpm"
    data_type: u16
    range: { min: 0, max: 15000 }
  - id: 2
    name: "Throttle_Position"
    unit: "%"
    data_type: f32
    range: { min: 0.0, max: 100.0 }

The Solver

Beta

pitgun-solver is the distributed compute layer. Compiled to WebAssembly via wasm-bindgen, it runs Monte Carlo simulations in browser-based Web Workers, aggregating results back to the mesh.

Key Types
  • RiskAnalysisRequest — Input: base config, lap count, scenario count, risk factors
  • RiskFactors — Tire degradation variance, rain probability, safety car chance
  • SimulationResult — Output: average time, success probability, tire failure rate
  • RaceStrategySolver — WASM entrypoint for solve_strategy()
WASM Compilation
$ wasm-pack build crates/pitgun-solver \
  --target web --out-dir pkg

Deploy the .wasm module to browsers. Each client becomes a volunteer compute node.

Monte Carlo Flow

The Solver receives a JSON-serialized RiskAnalysisRequest, runs N scenarios with randomized risk factors (rain, tire wear, safety car), and returns aggregated statistics. Each scenario is CPU-bound—designed to run in a dedicated Web Worker thread.

RiskAnalysisRequest
{
  "base_config": { /* CanonicalConfigV1 */ },
  "laps": 58,
  "scenarios_count": 10000,
  "risk_factors": {
    "tire_degradation_variance": 0.15,
    "rain_probability_per_lap": 0.02,
    "safety_car_chance": 0.05
  }
}
SimulationResult
{
  "average_total_time": 5220.45,
  "success_probability": 0.85,
  "tire_failure_rate": 0.02,
  "strategies_histogram": [
    /* distribution buckets */
  ]
}
JavaScript Integration
// In a Web Worker
import init, { RaceStrategySolver } from './pitgun_solver.js';

await init();
const solver = new RaceStrategySolver();
const result = solver.solve_strategy(JSON.stringify(request));
postMessage(JSON.parse(result));

Authority

WIP

The pitgun-authority service is the governance layer—designed to sign configuration manifests, enforce rate limits, and provide cryptographic proof of simulation authenticity. Currently in foundational development.

Design Intent
  • Manifest Signing: Cryptographically sign pipeline.yaml and tuning policies.
  • Tuning Limits: Enforce parameter bounds (e.g., max RPM, min tire pressure).
  • Auditability: Prove that result A derived from signed Config B.
  • Rate Limiting: Capability-based access control per client.
Policy Example
# policies/tuning.v1.yaml
version: v1
limits:
  max_engine_rpm: 15000
  min_tire_pressure_kpa: 140
  max_ers_deployment_kj: 4000
signed_by: authority.pitgun.io

Engine F1

Reference Impl

pitgun-engine-f1 is the reference physics implementation—a deterministic simulation engine for motorsport telemetry. It provides the Data Plane that the Solver uses for Monte Carlo lap simulations.

Key Features
  • PhysicsSource: Synchronous lap simulation without async overhead.
  • CanonicalConfigV1: Shared configuration contract with the Solver.
  • Deterministic: Same inputs → same outputs, enabling reproducible analysis.

Deployment

Ops

Deploy the full mesh with Docker Compose, or run individual services with Cargo. The docker-compose.yml in the repository root orchestrates Gateway + Authority.

Docker Compose
$ docker-compose up -d pitgun-gateway
$ docker-compose logs -f

# Scale solver nodes
$ docker-compose up -d --scale solver=4
Cargo (Development)
# Terminal 1: Gateway
$ cargo run -p pitgun-gateway --release

# Terminal 2: Replay data
$ cargo run -p pitgun-replay -- \
  --target 127.0.0.1:8080 \
  --input datasets/session.csv
Environment Variables
  • RUST_LOG=info,pitgun_gateway=debug — Tracing filter
  • PITGUN_GATEWAY_BIND=0.0.0.0:8080 — Bind address
  • PITGUN_GATEWAY_DATA_DIR=./data — Storage path
  • PITGUN_GATEWAY_ALLOW_NON_LOOPBACK=1 — External access

Replay Tool

Stable

pitgun-replay is a CLI tool for injecting historical CSV data into the Gateway, enabling testing and development without live data sources.

Basic Usage
$ cargo run -p pitgun-replay -- \
  --target 127.0.0.1:8080 \
  --input datasets/telemetry/nEngine.csv
CLI Options
  • --target <addr> — Gateway address
  • --input <file> — CSV file to replay
  • --rate <hz> — Playback speed (default: realtime)
  • --loop — Loop continuously
CSV Format

CSV files should have a timestamp_us column and parameter columns matching the Registry. The replay tool auto-detects columns and maps them to parameter IDs via name lookup.
